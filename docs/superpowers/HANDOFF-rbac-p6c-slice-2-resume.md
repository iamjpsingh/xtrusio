- 12 of 14 new+existing tests in `test_platform_audit_log.py` + `test_workspace_audit_log.py` PASSED.
- 2 FAILED on the first attempt — both were the "hard-deleted actor" tests. The underlying cause was that `rbac_audit_log.actor_auth_user_id` has FK `ON DELETE SET NULL` (per migration 0006), so when we hard-delete the auth.user the audit row keeps its identifying fields but loses the actor pointer — the original assertion `actor_auth_user_id == actor AND actor_email is None` can never be true. Additionally, the workspace test's tenant `created_by` FK is `ON DELETE RESTRICT`, so we can't delete the actor who created the tenant.

**I fixed both tests** before stopping (you interrupted me right as I was about to re-run pytest):

- Platform test now identifies the orphaned audit row by a `sentinel_target` UUID instead of the (now-null) `actor_auth_user_id`, and asserts `actor_auth_user_id is None AND actor_email is None`.
- Workspace test now creates a **secondary actor** that authors the audit event but does NOT own the tenant, then hard-deletes only the secondary actor. Same sentinel-target lookup pattern as above.
